# CONTEXT.md — บริบทธุรกิจและที่มาของการตัดสินใจ

ไฟล์นี้ตอบคำถาม "ทำไมระบบถึงเป็นแบบนี้" — อ่านคู่กับ ARCHITECTURE.md (ซึ่งตอบ "เป็นแบบไหน")

## ธุรกิจ

**Be Connected** เป็นบริษัทไทยให้บริการติดตั้งและพัฒนา software:
Solar Cell, CCTV, ANPR, Smart Home, IoT, network infrastructure
ทีมหลักคืองาน field: Engineer ออกไซต์ติดตั้ง/ซ่อม/บำรุงรักษา งานหนึ่งกินเวลา 1–5 วัน และ**คร่อมสัปดาห์/เดือนบ่อย**

## ปัญหาที่ module นี้แก้

- Engineer ไม่มีที่วางแผนงานล่วงหน้าเป็นระบบ / เช็คงานประจำวัน / สรุปว่าวันนี้เสร็จอะไร
- CEO ไม่เห็นภาพรวมว่าใครทำอะไรอยู่ที่ไหนในแต่ละวัน
- เมื่องานเลื่อน ไม่มีการบันทึก "เหตุผล" อย่างเป็นระบบ → วิเคราะห์ย้อนหลังไม่ได้

## Scope ที่ตกลงกัน (สำคัญ — เคย reset scope มาแล้ว 1 รอบ)

เดิมออกแบบเป็น ERP เต็ม 12 models (รวม inventory, ยืมอุปกรณ์, ลงเวลา, ลางาน, notification)
**ตัดสินใจ reset เหลือ module เดียวก่อน** เพื่อ iterate เร็ว — schema ERP เต็มเก็บไว้เป็น reference
(`schema_merged.prisma`) จะหยิบกลับมาทีละ module เมื่อถึงเวลา

Scope ปัจจุบัน:

| Role | เห็นอะไร | ทำอะไรได้ |
|---|---|---|
| ENGINEER | เฉพาะงานตัวเอง | สร้าง/แก้แผน, กดเริ่มงาน, กดจบงาน |
| CEO | ปฏิทินรวมทุกคน | **ดูอย่างเดียว** (ไม่สร้าง ไม่มอบหมาย ไม่แก้) |

หน้าจอฝั่ง Engineer มี 4 ส่วน และ**ทั้ง 4 ส่วนเป็นแค่ 4 มุมมองของ table เดียว (WorkPlan)**:

1. **ปฏิทิน** — WorkPlan ทั้งเดือน (query แบบ interval overlap)
2. **แผนงาน** — form สร้าง/แก้ WorkPlan (แก้ได้เฉพาะแผนตัวเองที่ยังไม่เริ่ม)
3. **สิ่งที่ต้องทำ** — WorkPlan ที่วันนี้อยู่ในช่วง + งานค้างจากวันก่อน + ปุ่มเริ่ม/จบงาน
4. **สรุป** — นับ status ประจำวัน + รายการจัดกลุ่มตาม status + delay reasons

ข้อ 1–2 ใช้ query `workPlan.list` (window รายเดือน) / ข้อ 3–4 ใช้ `workPlan.todo`
ซึ่งเห็นงานค้างข้ามเดือนที่ window รายเดือนมองไม่เห็น (เหตุผลใน ARCHITECTURE.md)

ปฏิทินรวมของ CEO ใช้ 3 เทคนิคร่วมกัน: สีประจำคน (`user.color`) + filter รายคน + แสดงแบบย่อ (+N รายการ)

## การตัดสินใจ product ที่ผ่านการถกแล้ว

| การตัดสินใจ | เหตุผล / ทางที่ไม่เลือก |
|---|---|
| ปฏิทินแสดงเฉพาะ WorkPlan | เคยพิจารณาแสดง ลา/ยืมรถ/กิจกรรม ด้วย → ตัดออกจาก scope, เพิ่ม field `type` ทีหลังได้โดยไม่พังโครง |
| WorkPlan : tracking = **1:1** | เคยเป็น 2 tables (WorkPlan + Todo) → ยุบเพราะ 1:1 และ Todo มี field denormalize ซ้ำ (ขัด single source of truth) |
| แผนเก็บเป็น**ช่วงวัน** (1 แผน = 1 แถว) ไม่แตกแถวรายวัน | ทุก field เป็นของทั้งแผน (`actStart`/`actEnd`/delay reason มีชุดเดียว — แตกรายวันจะ denormalize ซ้ำ), แก้/เลื่อนแผน = update แถวเดียวไม่ต้อง insert/delete แถวให้ atomic, ปฏิทินหาแผนคร่อมเดือนด้วย interval overlap query (index `(startDate, endDate)`) ส่วนการโชว์ pill ซ้ำทุกวันเป็นหน้าที่ของ view ตอน render — ถ้าอนาคตต้องการข้อมูลรายวัน (daily check-in) ให้เพิ่มตารางลูกผูก `workPlanId` ไม่ใช่แตกแถว WorkPlan |
| CEO ไม่มอบหมายงาน | ทำให้ไม่ต้องแยก `created_by`/`assigned_to` — เจ้าของแผน = คนสร้าง = คนทำ เสมอ |
| `actStart`/`actEnd` แยก column | เก็บเป็น timestamp เต็มเพื่อวิเคราะห์ KPI ย้อนหลังได้ (แผน vs จริง) |
| บังคับ delay reason | ข้อมูล "ทำไมงานเลื่อน" คือ value หลักของ module นี้ต่อ CEO |
| Job ID ระบบรันให้ — user ไม่กรอก | กันเลขชน/รูปแบบเพี้ยนจากการกรอกมือ (รันจาก Postgres sequence) — พอมี Job table ค่อยย้ายเลขรันไปที่นั่น |

## ผู้ใช้จริง

- Engineer ใช้จากมือถือที่หน้างานเป็นหลัก → UI ต้อง mobile-first
- ภาษาใน UI และ error message: **ไทย**
- Timezone เดียวทั้งระบบ: Asia/Bangkok (ICT, UTC+7) — ไม่มีแผนรองรับหลาย timezone
